# AGENTS.md

This file provides guidance to coding agents working in this repository.

## Project Overview

This project generates a procedural hex map with Wave Function Collapse (WFC) and renders it with Babylon.js. The WFC solver is written in Rust, compiled to WebAssembly, and executed inside a web worker so map generation does not block the UI thread.

## Tooling And Prerequisites

The repo expects:

- `pnpm` for JavaScript workspace commands
- Node.js 22 via `mise.toml`
- stable Rust
- the `wasm32-unknown-unknown` target
- `wasm-bindgen` and `wasm-opt` for producing browser-ready WASM output

Use the repo-managed setup when possible:

```bash
# Install Node/Rust versions from mise.toml
mise install

# Add the WASM target
mise run wasm-target

# Install wasm-bindgen-cli and wasm-opt
mise run wasm-tools
```

Manual fallback:

```bash
rustup target add wasm32-unknown-unknown
cargo install wasm-bindgen-cli@0.2.100
cargo install wasm-opt --version 0.116.1
```

## Recommended Workflows

### TypeScript / Rendering Changes

Use `pnpm run dev` for the normal edit-refresh loop. For TypeScript-only changes, `pnpm run check:types` is usually the fastest validation step.

### Rust / WASM Changes

After editing anything under `packages/wfc/src/`, rebuild the generated WASM assets with `pnpm run build:wasm:dev`. Before finishing, prefer `cargo test -p wfc-core` and a release rebuild via `pnpm run build:wasm:release` or `pnpm run build`.

Do not hand-edit files in `packages/wfc/wasm/`; they are generated by `scripts/build-wasm.mjs` via `cargo build`, `wasm-bindgen`, and, for release builds, `wasm-opt`.

### Tile Definition Changes

Tile definitions are duplicated across Rust and TypeScript:

- `packages/wfc/src/tile.rs`
- `packages/types/src/tile-def.ts`

When changing tile names, meshes, edge data, weights, or elevation metadata, update both files in the same change and run `pnpm run check:tiles`. `pnpm run check` is the safest final validation because it also runs the Rust test suite and TS type checks.

### Pre-Handoff

Unless the task is documentation-only, prefer `pnpm run check` before handing work back. That is the closest thing this repo has to a full local CI pass.

## Architecture

### Monorepo Structure

- `packages/app/` - application entrypoint consumed by Vite
- `packages/render/` - Babylon.js scene setup, tile pooling, and rendering logic
- `packages/types/` - shared TypeScript types and tile definitions (`@hex/types`)
- `packages/wfc/src/` - Rust WFC core compiled to the `wfc-core` WASM crate
- `packages/wfc/ts/` - TypeScript bridge, worker, and worker protocol for `@hex/wfc`
- `packages/wfc/wasm/` - generated JS/WASM artifacts emitted by the build script
- `scripts/build-wasm.mjs` - Rust-to-WASM build pipeline
- `scripts/check-tiles.mjs` - cross-checker for Rust and TS tile-definition parity

### WFC Core

The solver in `packages/wfc/src/solver.rs` uses trail-based backtracking:

1. Pick the cell with the lowest entropy.
2. Collapse it to a weighted random state.
3. Propagate adjacency constraints to neighbors.
4. Backtrack when a contradiction is reached.

Important Rust files:

- `packages/wfc/src/api.rs` - `wasm-bindgen` API exposed to JavaScript as `WfcEngine`
- `packages/wfc/src/solver.rs` - single-grid WFC solver
- `packages/wfc/src/multi_grid.rs` - 19-grid orchestration, global shared cell map, retries, and local re-solves
- `packages/wfc/src/placement.rs` - post-solve placement generation
- `packages/wfc/src/hex.rs` - cube-coordinate helpers and neighbor math

### TypeScript / Worker Layer

`WfcBridge` in `packages/wfc/ts/bridge.ts` creates a module worker from `packages/wfc/ts/worker.ts`. The worker imports `packages/wfc/wasm/wfc_core.js`, initializes the WASM module, and proxies `solveGrid`, `solveAll`, and placement generation back to the main thread through a small message protocol.

The grid-index ordering is a cross-language contract shared by:

- `packages/wfc/src/multi_grid.rs`
- `packages/wfc/ts/grid-positions.ts`

If grid ordering changes in one place, update the other.

### Data Flow

1. The app creates the Babylon renderer and a `WfcBridge`.
2. The worker initializes the WASM `WfcEngine`.
3. `solveAll()` solves a radius-2 hex-of-hex layout (19 grids total).
4. A global map shares boundary cells across grid solves for edge consistency.
5. Placement generation runs after tile collapse succeeds.
6. The renderer visualizes solved tiles and generated placements.

## Tile System

Tiles have 6 directional edges in hex order (`NE`, `E`, `SE`, `SW`, `W`, `NW`). The current edge types are:

- `Grass`
- `Water`
- `Road`
- `River`
- `Coast`

Elevation is not a separate edge type. Slopes and cliffs are represented by `high_edges` / `highEdges` plus `level_increment` / `levelIncrement`.

Other important rules:

- Rust tile IDs are positional; the vector index in Rust and the array index in `TILE_LIST` in TypeScript must stay aligned.
- `scripts/check-tiles.mjs` compares names, meshes, edges, weights, `preventChaining`, elevated edges, and level increments across both languages.
- The shared hex direction order matters for adjacency and for the tile-sync script.
